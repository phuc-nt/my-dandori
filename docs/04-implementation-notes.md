# Implementation Notes — Dandori MVP

> Kiến trúc THẬT sau khi build (260702), quyết định chốt, giới hạn đã biết. Bổ sung cho [01-product-vision.md](01-product-vision.md) / [03-features.md](03-features.md) — không thay chúng.

## Kiến trúc

Một binary `dandori` (Go 1.26, pure-Go, no CGO), một file SQLite (WAL) tại `~/.dandori/dandori.db`.

```
cmd/dandori            entry
internal/config        defaults → YAML → env → .env (env thắng .env)
internal/store         SQLite + migrations (PRAGMA user_version) + testseed
internal/capture       hook ingest · transcript parse · attribution · watcher
internal/govern        guardrail engine · budget · gate · audit hash-chain · kill
internal/learn         4 metric · grade percentile · ROI · leaderboard · provenance · quality gate
internal/integrations  guard (DRY_RUN/AGENT_WRITE_DISABLED) · jira · slack · ghub
internal/web           chi + html/template + go:embed + HTMX console
internal/cli           cobra commands + serve workers
```

### CAPTURE — cách hoạt động thật

- `dandori init` merge 4 hooks vào `.claude/settings.json` (idempotent, giữ hooks sẵn có) + ghi `.dandori-agent` (agent/project attribution).
- `dandori hook <event>` đọc JSON stdin của Claude Code. **Fail-open**: lỗi nội bộ → stderr + exit 0, không bao giờ phá phiên người dùng.
- Token/cost: parse transcript JSONL (`~/.claude/projects/<cwd-encoded>/<session>.jsonl`), dòng `type=assistant` → `message.usage`, **dedup theo `message.id`, full-reparse + SET** (không cộng dồn incremental) → không thể đếm đôi. Reconcile throttle 10s/run ở post-tool, chốt ở Stop.
- Watcher (C7): quét transcript mtime mới, session chưa có run → tạo `source=watcher`; đã có → chỉ reconcile usage. Chạy trong serve (ticker 60s) + `dandori watch`.
- Cost = tokens × bảng giá trong config (`pricing`, USD/MTok, prefix-match model id, fallback `default`). Giá lệch thực tế → sửa config, không sửa code.

### GOVERN — thứ tự check (cố định)

`kill switch → sandbox (G2) → block rules (G1) → secrets/PII (G1.5) → budget (G3) → gate (G4)` — hit đầu tiên thắng. **Fail-close** khi engine không eval được rule (ngược với capture). Mọi decision → event + audit entry (`hash = sha256(prev||ts||actor||action||subject||detail)`, `dandori audit verify` walk chain).

- Gate G4: tạo approval pending, poll DB 2s tối đa `gate_wait_seconds` (30s; hook timeout cài 40s). Approve trong lúc chờ → allow + consume; timeout/reject → deny kèm id. Web và Slack cùng quyết: **first-writer-wins** (UPDATE … WHERE status='pending'); race SELECT-then-INSERT đóng bằng partial unique index `approvals(run_id,action) WHERE status='pending'` (migration 019) — conflict thì re-select thay vì lỗi, đảm bảo 1 approval/action đang pending.
- Sandbox G2 chỉ enforce cứng cho Write/Edit/NotebookEdit; path token trong Bash là heuristic (miss thì lọt — block rules đỡ các case nguy hiểm). Giới hạn ghi nhận, chấp nhận MVP.
- **G1.5 secret/PII guardrail** (`internal/govern/secrets.go`): scan `Command` + `Content` (Write `content`/Edit `new_string`/NotebookEdit `new_source`) — KHÔNG scan Paths. Secret strict-set (`sk-`, `xox`, `ATATT`, `ghp_`, AKIA, PEM private-key header, Bearer non-env-ref) → **Deny**, message chỉ nêu loại (không echo giá trị). PII (thẻ Luhn-valid ≥1, hoặc ≥5 email) → **Gate**, action/reason mask trước khi ghi approval — **bỏ qua band** (trusted không skip PII gate, khác gate rule thường). Cap scan 512KB đầu+đuôi; **giới hạn đã biết**: literal regex, không bắt secret base64/split-token/env-expansion runtime. `redact.String` (ingest+export) là **superset** của mọi pattern Deny/Gate — test khoá bất biến `redact.String(x)!=x`. **Central mode chỉ chạy nửa secret-Deny** (regex thuần, không cần DB) trong `PolicySnapshot.Evaluate`; PII-gate cần `findOrCreateApproval`+wait nên **local-only**, không chạy trên snapshot. Cờ tắt: `secrets_guard_enabled: false` (mặc định bật).

### LEARN — proxy và giới hạn

| Metric | Công thức | Giới hạn proxy |
|---|---|---|
| Acceptance | 1 − rejected_edits/edits | chưa có revert-detection GitHub; reject signal chỉ từ tool_result ok=0 + guardrail block |
| Success | done/ended; có task_key → status Jira | run tương tác không link Jira dễ 100% |
| Autonomy | 1 − intervened_runs/runs (ask hoặc >1 user msg) | phiên interactive của người luôn bị tính intervened — đúng định nghĩa "supervised" nhưng làm autonomy=0 hàng loạt |
| Reliability | 1 − mean(err, block, kill rates) | — |

Grade: percentile composite trên fleet, A≥p80…F<p20; fleet <5 → static band + cờ `uncalibrated` (dấu `*`). ROI: 3 bucket loại trừ lẫn nhau (failed/killed → flagged → phần acceptance của clean) — không đếm đôi. Mọi metric trả `Formula` + run/event ids → `/provenance` drill về raw.

## v4 (260702, plan `260702-2040`) — Executive: sản phẩm cho CEO nhiều team

Nâng Dandori từ harness một-người thành sản phẩm điều hành cho CEO công ty nhiều đội. Toàn bộ đã qua red-team plan review (4 CRITICAL + 5 HIGH vá trước khi code).

- **Central mode (P1):** hook máy dev parse transcript **tại chỗ** và POST **chỉ số liệu đã redact** (`internal/ingest/derive.go`) — server KHÔNG bao giờ mở transcript của máy khác. **Hai listener một binary:** console `127.0.0.1:4777` (originGuard, không auth — không bao giờ public) + ingest listener riêng `0.0.0.0:4778` (Bearer token, `crypto/subtle`, không có route console). Idempotency: mỗi event 1 ULID + UNIQUE index → spool replay không đếm trùng. Pre-tool GOVERN chạy **local** trên máy dev qua policy snapshot cache (không round-trip/call → không nghẽn writer đơn); mất cache + server chết → chỉ chặn lệnh ghi (fail-closed hẹp). Read-only pool tách khỏi writer. `dandori connect` (token vào file 0600), `dandori relay`, `dandori team`.
- **Operators + Teams (P1):** `operators` (người điều khiển, id = `username@hostname` lấy từ token principal), `teams`/`team_members` (gồm operator + agent). Team là khái niệm hạng nhất để so sánh hiệu quả giữa đội.

## v14 (260708, plan `260708-2156`) — Govern hardening

Bốn nâng cấp GOVERN nhặt từ so sánh Omnigent (`plans/reports/comparison-260708-2147-*`), qua red-team 3 reviewer (10 finding, 4 Critical) trước khi implement. Thứ tự dưới: fail-closed contract → G1.5 secret/PII → G3 downgrade → G5 risk-score.

### Fail-closed contract + break-glass (`internal/govern/contract.go`)

`CheckFailureModes` là nguồn sự thật duy nhất: mỗi check-point (engine chain + dispatch layer trong `internal/cli`) khai báo `FailClosed`/`FailClosedMutating`/`FailOpen`. Quy tắc: check chạy TRƯỚC side-effect thì fail-closed (deny) khi không eval được; capture-only thì fail-open. Đã vá 4 nhánh fail-open trên đường hook local (red-team): `openStore` lỗi → deny mutating (đọc vẫn chạy); `ing.PreTool` lỗi tách khỏi verdict (verdict chạy độc lập capture); config-load lỗi → cảnh báo stderr + fallthrough local; input-decode giữ fail-open có chủ đích (chưa parse được ToolName thì không có gì để deny — map ghi đúng `FailOpen`, không nói dối).

- **Break-glass `DANDORI_GOVERN_FAIL_OPEN=1`** (env, đọc TRƯỚC khi mở DB): khi DB hỏng/lock/disk-full khiến engine-unavailable deny bricking mọi agent, đặt env này để tạm fail-open (log rõ mỗi lần active). Env chứ không config-file vì file có thể nằm trong DB đang hỏng. **Chỉ phủ local-DB mode** — central mode (server unreachable + không cache snapshot) chưa có lối thoát tương đương; đó là gap đã biết, chấp nhận vì central không phải mode mặc định.
- **Seed block rule bảo vệ `~/.dandori/`** (migration `018_protect_dandori_home.sql`): config re-read mỗi hook + Bash không bị sandbox chặt → agent có thể `echo secrets_guard_enabled:false >> ~/.dandori/config.yaml` tự tắt guardrail, hoặc xoá DB để kích fail-open. Rule chặn Bash đụng `~/.dandori/` (bắt các cách viết `~`/`$HOME`/`/Users/x`/`/home/x`, không false-positive `~/.dandori-backup`) — cùng tier seed rule chặn `.env`. Regex block chỉ bắt cách viết thường; kẻ tấn công quyết tâm vẫn né được (inherent — G2 sandbox mới là backstop cho Write/Edit).

**E2E verify (binary thật, không unit):** drive `dandori hook pre-tool` qua stdin→verdict JSON trên DB+config cô lập. 14/14 pass: G1.5 secret deny (message "value withheld"), Bearer-$ENV allow, PII card → approval row tạo rồi headless timeout→deny, `~/.dandori/` block đủ 3 cách viết (không FP `~/.dandori-backup`), break-glass flip DB-lỗi deny→allow, audit chain verify intact. **Tính chất đáng chú ý:** G1.5 fire **độc lập** G1/G2 — trên máy đã đặt `block_enabled: false`/`sandbox_enabled: false`, secret/PII guardrail vẫn chặn (cờ `secrets_guard_enabled` riêng), nên máy đã-gỡ-guardrail vẫn được bảo vệ khỏi rò secret.

### G3 budget breaker

Red-team pass trên G3 budget breaker: vượt trần đổi từ hard-stop toàn bộ sang **downgrade-gate** — chỉ Deny khi run đang chạy model đắt (kèm gợi ý đổi `/model`), model rẻ chạy tiếp. `budget.mode: hard` phục hồi hard-stop cũ cho operator muốn giữ nguyên trạng.

- **`Budget.Mode`/`ExpensiveModels`/`NullAllowCap`** (`internal/config/config.go`): `""`/`"downgrade"` (mặc định) hay `"hard"`. `ExpensiveModels` nil → default `["opus","fable"]` — chốt từ pricing table thật (`claude-opus-4-8`/`claude-fable-5` output=25 so với sonnet=15/haiku=5, KHÔNG có `gpt-5` trong pricing hay fleet). List rỗng tường minh `[]` = tắt hẳn trần cứng, chỉ còn cảnh báo % — giữ semantic Omnigent. `NullAllowCap` ≤0 → mặc định 20.
- **`checkBudget` downgrade path** (`internal/govern/budget.go`): vượt 100% → `runModel(tc.RunID)` (point-query `runs.model`, lỗi query = fail-closed Deny); model rỗng (chưa reconcile) → `nullAllowGate`; match expensive (substring case-insensitive) → Deny + message nêu tên model + `/model`; ngược lại → Allow + event `budget_downgrade_allow` (dedup qua events-query theo run+tháng, KHÔNG settings-key-per-run — tránh leak cardinality).
- **NULL-model chống evasion (`nullAllowGate`):** hai lớp — (1) agent-scoped fallback: run trước gần nhất của agent này trong tháng nếu là model đắt → NULL hiện tại coi như đắt, Deny (chặn agent restart session để giữ model NULL); (2) cap `null_allow:{agent}:{month}` (settings key, bounded — một dòng/agent/tháng, KHÁC với tombstone-per-run) — vượt cap → Deny dù chưa có lịch sử đắt.
- **Central mode giữ hard-stop, có chủ đích** (`policy_snapshot.go`): snapshot chỉ mang `BudgetExceeded bool` toàn cục, không có model/scope per-run/DB — không đủ dữ liệu để chạy downgrade-gate. Comment tại `Evaluate` khai rõ divergence; test `TestSnapshotBudgetBranchUnchanged` khoá hành vi.
- **Giới hạn đã biết (không vá trong v14):** transcript-forgery — agent Write/Edit được vào `~/.claude/projects/` (memory dir cho phép ghi, xem `sandbox.go` allowlist) nên có thể tự sửa transcript để giả model rẻ; downgrade-gate tin `runs.model` là nguồn thật, không xác minh chữ ký/hook-input. Runtime non-Claude (không có `TranscriptPath`, vd Codex wrap) chỉ set `model` ở `FinalizeRun` cuối run — NULL suốt toàn bộ mid-run, luôn đi qua `nullAllowGate` cho tới khi run kết thúc.

Test: `internal/govern/budget_test.go` (mới) + regression `TestBudgetBreaker` (engine_test.go) cập nhật seed model để giữ đúng ý nghĩa "vượt trần + model đắt = Deny". `go vet ./...`, `go test ./...` xanh toàn repo.

### G5 — Risk-score guardrail (`internal/govern/risk.go`, `risk_score.go`)

Guardrail "có trí nhớ": run tích điểm rủi ro theo hành vi gần đây, vượt ngưỡng thì tool canh gác (mặc định chỉ `Bash`) leo thang. Thiết kế gốc là **ratchet tự khuếch đại** (điểm ăn chính `guardrail_block` insert vô điều kiện của chính engine, kể cả Allow/`engine_error`/`risk_gate`) — bản này sửa bằng 3 lớp:

- **Default `mode: log`, không `gate`.** Vượt ngưỡng chỉ emit event `risk_would_gate` (payload nêu score/threshold), KHÔNG chặn tool call. Chỉ `mode: gate` mới leo thang thật qua `findOrCreateApproval`+`waitDecision` (dùng lại máy approval của G4, KHÔNG qua nhánh band-skip của `checkGate` — trusted band không miễn G5, khác G4).
- **Sliding window, không tích luỹ cả run.** `RiskScore(runID)` chỉ nhìn `WindowN` (default 40) event gần nhất của run (`events(run_id,ts)` index có sẵn, không cần migration). Base = tổng `tool_points` (default Bash=2, Write/Edit/NotebookEdit=1, WebFetch=5) của các event `kind='tool_use'` trong window. Cửa sổ thời gian của phần "denial thật" khớp với event cũ nhất trong window (`ts >= events[0].ts`) — cùng một khái niệm "cửa sổ" áp cho cả hai nguồn, không phải hai window lệch nhau.
- **Self-exclusion — discriminator qua `audit_log.action`, không phải `events.kind`.** `record()` map MỌI action (trừ `gate_decision`) vào `kind='guardrail_block'` nên không thể phân biệt Allow/lỗi/risk_gate/block-thật chỉ bằng `events.kind`. Denial điểm CHỈ đếm từ `audit_log` (subject=run_id) với action ∈ `{guardrail_block, sandbox_block, secrets_block}` — loại trừ tuyệt đối `engine_error`, `budget_block`, `risk_gate` (chính G5 escalate), và mọi Allow. `audit_log` không có index theo subject nhưng là bảng nhỏ append-only (xác nhận từ `001_init.sql`) — scan lọc theo run là chấp nhận được, không thêm migration.

Config `RiskScore` (`internal/config/config.go`): `mode` (""/"log" mặc định, "gate" opt-in), `threshold` (≤0 → mặc định 100, chỉ có ý nghĩa ở gate mode), `window_n` (≤0 → 40), `tool_points` (nil → default map trên), `denial_points` (≤0 → 25), `guarded_tools` (nil → `["Bash"]`).

**Central mode không có G5** — cố ý (`policy_snapshot.go` giữ nguyên logic, chỉ thêm comment): risk score cần `events`+`audit_log` local và máy approval, snapshot offline không có DB để query. `TestSnapshotEvaluateHasNoRiskBranch` khoá hành vi không đổi.

Badge run-detail (`internal/web/handlers_runs.go` + `templates/pages/run_detail.html`): tính `RiskScore(runID)` read-only tại render, best-effort (lỗi query chỉ ẩn badge, không fail trang). Ẩn hoàn toàn khi score=0 (nhất quán với các stat khác trên trang).

### Audit hash-chain external anchor (`internal/govern/audit_*.go`)

Đóng gap v12: chain hash tự-nhất-quán vẫn có thể bị rebuild toàn bộ từ đầu bởi ai có quyền ghi thẳng SQLite. Giờ có 4 lớp độc lập, mỗi lớp chặn một attack riêng:

- **Canonical hash chống delimiter-shift** (`audit_canonical.go`): field length-prefix (8-byte big-endian) trước khi sha256, thay `strings.Join(...,"|")` cũ — join cũ cho phép hai bộ field khác nhau nhưng cùng byte string nếu ký tự `|` di chuyển giữa nội dung field và vị trí delimiter (test dựng collision thật `TestCanonicalHashResistsDelimiterShift`).
- **Ed25519 co-sign mỗi hàng** (`audit_sign.go`): key qua env `DANDORI_AUDIT_SIGNING_KEY` (base64, seed 32 byte hoặc full key 64 byte) — **không đưa vào YAML**, cùng pattern với `OPENROUTER_API_KEY`/`ATLASSIAN_*`. Không set key → hàng không ký, cảnh báo stderr một lần, hành vi cũ giữ nguyên (backward compat).
- **Monotonic signing** (`audit.go`, `first_signed_id` trong bảng settings): một khi chain có hàng đã ký, không hàng nào sau đó được phép "tụt" về unsigned — chặn kiểu rebuild xoá signature (`UPDATE audit_log SET signature=NULL`) mà hash chain tự nó không thấy gì sai.
- **Checkpoint ký ngoài DB** (`audit_checkpoint.go`): `{tip_id, tip_hash, ts, first_signed_id}` ký Ed25519, ghi file `docs/audit-checkpoints/<tip_id>.json` (field đổi tên từ `chain_len`→`tip_id` — đây là id của hàng tại thời điểm checkpoint, không phải COUNT(*)). Tự động **ngay khi signing bật lần đầu** (hàng ký đầu tiên, không đợi tới hàng 100) + mỗi 100 hàng sau đó + mỗi lần export compliance. Đây là lớp duy nhất bắt được xoá tail rows: các hàng còn lại vẫn tự-nhất-quán hash+signature, chỉ có checkpoint (nguồn ngoài DB) mới biết chain đã ngắn hơn trước.

**Checkpoint tự nó phải được verify chữ ký trước khi được tin** (`verifyAgainstCheckpoint` + `cp.VerifySignature(pub)` trong `Verify()`): một checkpoint file tồn tại nhưng chữ ký sai (forge, corrupt, ký bằng key khác) bị coi là tampering (`reason="checkpoint"`), **không bao giờ** fallback im lặng về "không có checkpoint". Đây là điểm mấu chốt nâng bar từ "ghi được file" lên "phải giữ private key ký".

`first_signed_id` nằm trong payload đã-ký của checkpoint — nên xoá marker `audit_first_signed_id` trong DB không xoá được sự thật signing đã bật: `Verify()` lấy floor là max(marker trong DB, `cp.FirstSignedID` của checkpoint hợp lệ mới nhất). **Key-required mode**: nếu signing key đang được cấu hình NHƯNG tail chain unsigned và không có checkpoint hợp lệ nào chứng minh signing từng tắt đúng cách → `Verify()` fail với `reason="signature"`, không im lặng báo intact — đóng khoảng trống dưới 100 hàng / trước lần export đầu tiên.

**Git-commit best-effort** (`gitCommitCheckpoint` trong `audit_checkpoint.go`): mỗi lần `WriteCheckpoint` chạy `git add` + `git commit` cho file checkpoint nếu `dir` nằm trong git work tree — best-effort như offsite copy, log lỗi ra stderr, không bao giờ fail lần ghi audit. Bỏ qua im lặng nếu không có `git` binary hoặc `dir` không phải git repo. **Đây là automation, không phải security boundary** — trên một máy đơn nơi signing key cũng sống, kẻ tấn công có cả key + quyền truy cập máy vẫn re-sign forged checkpoint và commit đè lên bản cũ. Phòng thủ thật với máy bị compromise hoàn toàn là: (1) checkpoint-signature-verification ở trên (phải có key mới forge được) + (2) bản copy nằm ở nơi kẻ tấn công không với tới được — remote git `push`, hoặc `DANDORI_AUDIT_CHECKPOINT_OFFSITE` trỏ vào storage khác quyền kiểm soát. Xem doc comment `DefaultCheckpointDir` trong code cho threat model đầy đủ.

`Verify()` trả `(brokenID, reason, err)` — `reason` ∈ `{"", "chain", "signature", "checkpoint", "truncated"}` phân biệt 4 lớp tấn công thay vì chỉ bool. CLI `dandori audit verify` in kèm hint theo reason. CLI mới: `dandori audit keygen` (sinh keypair), `dandori audit pubkey` (in public key để cross-check ngoài băng) — cả hai fingerprint (sha256 hex) qua `fingerprint()` hash **đúng một dạng chuẩn** (raw decoded bytes), trả lỗi tường minh nếu base64 decode fail thay vì âm thầm hash text base64 khi decode lỗi. HTTP `GET /ingest/audit-pubkey` (cùng bearer-token với `/ingest/*` khác) để máy khác lấy public key verify độc lập.

Gotcha đáng nhớ: `Verify()` ban đầu tự deadlock — `store.Store` cap `SetMaxOpenConns(1)`, mở cursor `rows` từ `st.DB.Query` xong mới gọi thêm một query khác (đọc `first_signed_id`) trong lúc cursor còn mở sẽ chờ vĩnh viễn connection mà chính cursor đang giữ. Fix: đọc hết settings/keys/checkpoint file **trước** khi mở `rows`. **CALIBRATION RULE (bắt buộc trước khi bật `mode: gate`):** threshold mặc định 100 chỉ là log-mode-safe placeholder. Trước khi đổi sang gate, operator PHẢI chạy query dưới trên events thật và đặt `threshold ≥ 3 × p99` của run bình thường — không phải quyết định tay:

```go
// Sketch — dùng govern.NewEngine(cfg, st).RiskScore(runID) cho từng run,
// gom lại và lấy p99/max trên population run bình thường (không seed demo).
rows, _ := st.DB.Query(`SELECT id FROM runs`)
var scores []int
for rows.Next() {
    var id string
    rows.Scan(&id)
    if s, err := eng.RiskScore(id); err == nil {
        scores = append(scores, s)
    }
}
sort.Ints(scores)
p99 := scores[int(float64(len(scores))*0.99)]
// threshold nên ≥ 3 * p99
```
